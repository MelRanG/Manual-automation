## DocOps AI의 유일한 영역 (경쟁자 Zero)

### 1. 답변 → 문서 역류 루프
```
기존: 문서 → AI 답변 → (끝. 문서 틀려도 계속 틀린 답변)
우리: 문서 → AI 답변 → 오류 제보 → AI 수정안 → 승인 → 문서 갱신 → 답변 개선
```
**왜 아무도 안 하나:** RAG 플랫폼은 "검색 품질"에 집중하고, 문서 관리 플랫폼은 "AI Q&A"에 집중함. 양쪽을 잇는 제품이 없음.

### 2. SR 완료 → 문서 자동 반영 추천
```
기존: SR 완료 → 개발팀 배포 → 문서 담당자가 기억나면 수동 업데이트 (대부분 까먹음)
우리: SR 완료 → AI가 영향 문서 분석 → 수정 전략 추천 → 담당자 승인 → 자동 반영
```
**왜 아무도 안 하나:** Jira ↔ 문서 시스템 사이의 연결이 기술적으로 어렵고, 비즈니스 로직이 도메인마다 다름.

### 3. 문서 신뢰도 정량화
```
기존: 문서가 오래됐는지 → 사람이 기억/추측
우리: 오류 제보 빈도, 마지막 검토일, 승인 대기 수, 조회 대비 불만족률 → 자동 계산
```
**왜 아무도 안 하나:** Guru가 2단계(verified/stale)만 함. 다차원 점수화는 데이터 모델이 복잡해짐.

## 비즈니스 임팩트

### 문서 오류가 기업에 미치는 비용

| 지표 | 수치 | 출처/근거 |
|------|------|-----------|
| 직원이 정보 검색에 쓰는 시간 | 주당 9.3시간 (전체 업무의 23%) | McKinsey, IDC |
| 오래된/틀린 문서로 인한 재작업 비율 | 전체 업무의 ~11% | PwC Knowledge Worker Survey |
| 문서 담당자 수동 업데이트 시간 | 주 5-8시간 | 현장 인터뷰 기반 추정 |
| 신입/전환 배치 온보딩 지연 | 평균 +2주 (문서 부정확 시) | SHRM 보고서 |

### DocOps 도입 후 예상 개선

| Before | After | 개선 |
|--------|-------|------|
| 문서 최신화율 30% | 80%+ | **+167%** |
| 오류 발견 → 수정 완료: 평균 14일 | 평균 2일 | **-86%** |
| SR 완료 → 문서 반영 누락률 60% | 10% 이하 | **-83%** |
| 문서 담당자 주간 수동 작업 8시간 | 1.5시간 | **-81%** |

### 핵심 ROI 문장
```
100인 기업 기준:
- 주당 절감: (100명 × 2시간 검색 절감) + (5명 담당자 × 6.5시간 절감) = 232.5시간/주
- 연간 환산: ~12,000시간 = 약 6 FTE 인건비 절감
```

## 심사위원 예상 질문 & 답변

### "Notion AI랑 뭐가 달라?"
> Notion AI는 문서에서 답변만 합니다. 답변이 틀려도 원본 문서가 수정되지 않습니다.
> DocOps AI는 사용자가 "틀렸다"고 알려주면, AI가 수정안을 만들고, 담당자가 승인하면, 문서가 업데이트되고, 다음부터 답변도 바뀝니다.
> 단방향 RAG vs 양방향 문서 생명주기 관리 — 카테고리가 다릅니다.

### "Glean이 이미 이거 하지 않나?"
> Glean은 100개 이상의 앱을 연결하는 "엔터프라이즈 검색 엔진"입니다. 검색 품질은 최고지만, 문서를 수정하는 기능은 없습니다. 오류를 발견해도 원본 시스템에 가서 직접 고쳐야 합니다.
> 우리는 검색보다 "문서가 항상 최신 상태인가"에 집중합니다.

### "Guru의 검증 워크플로우랑 뭐가 다른데?"
> Guru는 "이 카드가 30일 지났으니 다시 확인해"라는 시간 기반 만료입니다.
> DocOps AI는 "이 문서의 이 문장이 실제와 다르다"는 증거 기반 수정입니다.
> 시간이 지나면 전부 재검토 vs 문제가 발생한 부분만 정밀 수정 — 운영 비용이 완전히 다릅니다.

### "MVP에서 실제로 돌아가는 거 보여줄 수 있나?"
> 네. [데모 시나리오로 전환]
> 1. 문서 업로드 → 2. 질문/답변 → 3. 오류 제보 → 4. AI 수정안 → 5. 승인 → 6. 재질문 시 수정된 답변
> 이 전체 루프가 라이브로 동작합니다.
